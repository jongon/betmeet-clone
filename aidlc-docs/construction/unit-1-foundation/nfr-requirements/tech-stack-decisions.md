# Unit 1: Foundation — Tech Stack Decisions

## Context

The project stack is pre-selected: Next.js 16 (App Router), TypeScript, Tailwind CSS v4, shadcn/ui, Supabase (Auth + PostgreSQL + Storage), Prisma, pnpm, Vercel. The decisions below apply or configure this stack for Unit 1's specific requirements.

---

## Decision 1: Supabase Auth SSR Integration

**Choice**: `@supabase/ssr` package with Next.js App Router middleware and Server Components.

**Rationale**:
- Supabase Auth tokens are stored in `httpOnly` cookies via the SSR helpers, keeping them inaccessible to JavaScript (mitigates XSS token theft).
- Middleware validates and refreshes the session on every request, ensuring the session state is always current before Server Components execute.
- Consistent server/client auth state without waterfall requests.

**Implementation approach**:
- `src/lib/supabase/server.ts` — creates a Supabase client for Server Components and Server Actions using `createServerClient`.
- `src/lib/supabase/client.ts` — creates a browser Supabase client using `createBrowserClient` for Client Components.
- `src/middleware.ts` — refreshes the session on every route using `updateSession`.

---

## Decision 2: "Remember Me" Session Implementation

**Choice**: Customize the cookie `Max-Age` based on the "Remember me" checkbox at sign-in time.

**Rationale**:
- Supabase Auth does not natively expose a "remember me" option, but the SSR helpers allow passing custom cookie options.
- Without "Remember me": cookies are session cookies (no `Max-Age`). They are cleared when the browser is closed.
- With "Remember me": cookies are set with `Max-Age = 30 * 24 * 60 * 60` (30 days). The Supabase refresh token is stored with the same expiry.

**Implementation approach**:
- The sign-in Server Action accepts a `rememberMe: boolean` parameter.
- After calling `supabase.auth.signInWithPassword()`, the resulting session's refresh token expiry is set by passing custom cookie options to `createServerClient`.
- A utility `getSessionCookieOptions(rememberMe: boolean)` returns the appropriate `{ maxAge }` value.

## Decision 3: Profile Row Creation Strategy

**Choice**: Create the `Profile` row inside a Supabase Auth `on_auth_user_created` trigger (PostgreSQL trigger on `auth.users`), not in application code.

**Rationale**:
- A database trigger guarantees the Profile row is created atomically when `auth.users` is inserted, even if the app Server Action fails or the user closes the browser immediately after auth.
- Avoids a race condition where a user lands on a page with a valid session but no `Profile` row.
- Default values (random default avatar, `verification_status = UNVERIFIED`) are set in the trigger.

**Alternative considered**: Creating the Profile in the OAuth callback Server Action. Rejected because it can be skipped on network failure.

---

## Decision 4: Nickname Availability Check — Server Action + Debounce

**Choice**: Dedicated Server Action `checkNicknameAvailability(base: string)` called via client-side debounce (500 ms).

**Rationale**:
- Server Actions in Next.js App Router are the correct pattern for server-side data access from Client Components without exposing a REST endpoint.
- A 500 ms debounce prevents a query on every keystroke (critical for a shared DB at small scale).
- The action queries the `profiles` table using Prisma for the count of rows with the given `nickname_base`, returning `{ available: boolean }`.

**Note**: The availability check only validates that the base is usable. The final discriminator is generated atomically at submission time (WF-04) using database-level uniqueness enforcement.

---

## Decision 5: Avatar Upload — Direct Client Upload via Supabase Storage Signed URLs

**Choice**: Client uploads the file directly to Supabase Storage using a signed upload URL generated by a Server Action.

**Rationale**:
- Avoids routing the file binary through the Next.js server / Vercel Edge Function (reduces latency and Vercel function memory pressure).
- The Server Action generates a short-lived signed upload URL scoped to `avatars/custom/{user_id}/`. The client uploads directly.
- After upload, the client calls another Server Action `setAvatarFromUpload(storagePath)` to update the Profile row and delete the previous custom file.

**Replace strategy**: When `setAvatarFromUpload` is called and `Profile.avatar_source` is already `CUSTOM_UPLOAD`, the previous file at the stored path is deleted from Supabase Storage before saving the new path.

---

## Decision 6: Thumbnail Generation

**Choice**: Use Supabase Storage's built-in image transformation API (available on Supabase Pro plan) to generate display thumbnails on-the-fly via URL parameters.

**Rationale**:
- No separate thumbnail generation step needed at upload time.
- Thumbnails are generated on first request and cached by Supabase CDN.
- Avoids a dependency on `sharp` or an Edge Function for image processing.

**Thumbnail URL pattern**: `{storage_url}?width=64&height=64&resize=cover`

**Fallback**: If the project uses the Supabase Free plan (no image transforms), store a second copy of the file resized to 64×64 using a Vercel Edge Function at upload time.

---

## Decision 7: Input Validation — Zod Shared Schema

**Choice**: Define Zod schemas in `src/features/auth/schemas.ts` and `src/features/profile/schemas.ts`, used by both client-side form validation and Server Action validation.

**Rationale**:
- Single source of truth for validation rules (RULE-NICK-01, RULE-AVTR-03, password policy).
- Prevents client-server mismatch where the form accepts input that the server rejects.
- Consistent error messages between client and server.

**Key schemas**:
- `NicknameBaseSchema` — validates nickname_base format.
- `AvatarUploadSchema` — validates MIME type and file size (used in Server Action after receiving file metadata).
- `SignUpSchema` — validates email, password, confirmPassword.
- `ChangePasswordSchema` — validates currentPassword, newPassword, confirmNewPassword.

---

## Decision 8: Accessibility — shadcn/ui + Radix UI Primitives

**Choice**: Use shadcn/ui components (built on Radix UI) as the base for all Unit 1 UI, supplemented with custom ARIA attributes where needed.

**Rationale**:
- Radix UI primitives are WCAG 2.1 AA compliant by design: Dialog, Tabs, Select, and Alert components include correct focus management, keyboard handling, and ARIA attributes.
- Color contrast is enforced via Tailwind CSS v4 design tokens defined in `src/app/globals.css`.
- `aria-live` regions are added to form error containers to announce errors to screen readers.
- Avatar grid (custom) uses `role="listbox"` + `aria-selected` + keyboard arrow key handler.

---

## Decision 9: WCAG AA Color Contrast Enforcement

**Choice**: Define the color palette in `globals.css` CSS variables and validate contrast ratios during UI development using browser DevTools accessibility checker.

**Minimum contrast ratios**:
- Normal body text: 4.5:1 against its background
- Large text (≥ 18px or ≥ 14px bold): 3:1
- UI component borders and focus rings: 3:1
